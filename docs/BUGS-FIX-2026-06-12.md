# Bug Fixes — 2026-06-12

## 🐛 Bug 1: Upload de Excel falhava com "row-level security policy"

### Causa Raiz
A app enviava `user_id = auth.uid()` (UUID do Supabase Auth), mas a tabela `imports.user_id` é uma FK para `users.id` (UUID diferente gerado pela app). O trigger `handle_new_user` mapeia `auth.users.id` → `public.users.id` mas não havia um trigger em `imports`.

### Fix Aplicado
**Trigger `resolve_user_id()` em 5 tabelas:**
- `imports` ✅
- `purchase_orders` ✅
- `audit_logs` ✅
- `usage_events` ✅
- `favorites` ✅

A função resolve automaticamente `auth.uid()` → `users.id` no BEFORE INSERT. Se o user não existir, cria-o (self-healing).

## 🐛 Bug 2: Storage `ocr-uploads` também bloqueava

### Causa Raiz
A policy `ocr-uploads: admin all` verificava `app_metadata.role` em vez de `user_metadata.role`. O `app_metadata` é gerido só pelo Supabase (não pelo user), enquanto o `role` está em `user_metadata`.

### Fix Aplicado
- Storage policy `admin all` → `user_metadata.role = 'admin'`
- Nova policy `user upload own`: `foldername(name)[1] = auth.uid()` (user só pode fazer upload para a sua pasta)

---

## 🐛 Bug 3: `usage_events` INSERT policy exigia `user_id = auth.uid()`

### Causa Raiz
Mesmo com o trigger, a policy ainda verificava `user_id = auth.uid()` — o que falhava porque o trigger resolve para `users.id` (diferente).

### Fix Aplicado
Policy agora aceita ambos:
```sql
WITH CHECK (
  user_id = auth.uid()
  OR user_id = (SELECT id FROM users WHERE supabase_user_id = auth.uid() LIMIT 1)
)
```

## 🐛 Bug 4: `audit_logs` sem INSERT policy

### Causa Raiz
Tinha policies de SELECT, UPDATE, DELETE mas faltava INSERT.

### Fix Aplicado
Policy adicionada com `user_id IS NULL OR user_id = auth.uid() OR user_id = (resolved)`.

## 📋 Lições

1. **Sempre verificar FK vs `auth.uid()`** — o `users.id` ≠ `auth.uid()`
2. **`app_metadata` vs `user_metadata`** — policies devem olhar a metadata certa
3. **Triggers de resolução** evitam o problema do FK mismatch em qualquer tabela nova
