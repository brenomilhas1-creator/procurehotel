1 Visão Geral — 4 Pilares

A aplicação tem 4 componentes principais. Tudo é gratuito (free tier). O utilizador final só fala com o Vercel; o Vercel fala com o Supabase.

👤

Utilizador

Comprador num hotel. Usa o browser em desktop ou telemóvel. Faz login, pesquisa produtos, cria ordens.

▲

Vercel

Frontend Next.js 15. Build automático quando há push no GitHub. URL pública global com HTTPS e CDN.

⚡

Supabase

Banco de dados Postgres + Auth + Storage + Edge Functions (Deno). Tudo num só serviço. RLS protege os dados.

🐙

GitHub

Source code. Cada push para main dispara novo deploy na Vercel automaticamente.

Fluxo do utilizador: Browser → HTTPS → Vercel (página HTML+JS) → REST API → Supabase (dados + auth).
Fluxo de código: Programador → git push → GitHub → webhook → Vercel build + Supabase deploy.

2 Fluxo de Autenticação (Login)

Como o utilizador entra no sistema. Tudo gerido pelo Supabase Auth (GoTrue). O frontend NUNCA vê passwords em plain text — só hashes.

Segurança: O password NUNCA passa pelo Vercel — é enviado direto do browser para o Supabase via HTTPS. O Vercel só vê o JWT (que é assinado e verificável).

3 Fluxo de Pedido Rápido (CRUD)

O utilizador pesquisa um produto (autocomplete), escolhe, ajusta quantidade, e grava uma ordem. Tudo client-side + RLS.

RLS (Row Level Security): Cada query do Vercel para o Supabase envia o JWT no header. O Supabase valida automaticamente que o user pode ler/escrever aquela linha. Sem RLS, o user poderia ver ordens de outros hotéis.

4 Fluxo de Admin (Edge Function)

O admin cria novos utilizadores. Isto requer a service_role key (que NÃO está no frontend). Solução: Edge Function no Supabase.

Porquê Edge Function? A service_role key tem poder total (bypassa RLS). Se estivesse no frontend, qualquer pessoa a podia ler. Edge Function = código servidor sem servidor a gerir. Grátis até 500K invocations/mês.

5 Fluxo de Upload (Storage)

O utilizador carrega um ficheiro de preços (PDF/CSV/Excel). Vai para o Supabase Storage e fica em fila para processamento.

6 Fluxo de Deploy (CI/CD)

Cada push para o GitHub dispara deploys automáticos. Sem intervenção manual.

Tempo de deploy: ~1 minuto do push até o URL público estar atualizado. O Vercel faz cache do build, então alterações só de código são ainda mais rápidas (~30s).

7 Segurança — RLS em Detalhe

Cada query é validada pelo Postgres. Mesmo que o frontend mande um user_id de outro user, o RLS bloqueia.

8 Camadas de Defesa

Defense in depth — mesmo que uma camada falhe, as outras protegem.

🔐

Camada 1 — Autenticação

JWT ES256 com chave assimétrica. Refresh tokens rodáveis. Logout limpa localStorage.

🛡

Camada 2 — Autorização (RLS)

15 policies no Postgres. Cada query é validada. Admin vê tudo; user vê só os seus.

🌐

Camada 3 — Transport (TLS)

HTTPS forçado. HSTS max-age 2 anos. CSP no meta. Headers X-Frame-Options DENY.

🔒

Camada 4 — Storage (RLS)

Bucket privado. Cada user só vê os seus uploads. URL assinada temporária.

⚡

Camada 5 — Edge Function

service_role key NUNCA sai do Supabase. Validação extra: role==admin no JWT.

🔍

Camada 6 — Push Protection

GitHub bloqueia commits com secrets (sb_secret_, AWS keys, etc). Verified antes de push.
